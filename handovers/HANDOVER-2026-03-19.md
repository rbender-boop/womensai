### 2. Share Card + Auth-Gated Email Sharing (Commit 2)

**Problem:** No easy share affordance near the Best Answer. No user profile capture.

**Solution — Share card next to Best Answer:**
- `components/share-card.tsx` — positioned as right sidebar on desktop (sticky), horizontal strip on mobile
- Social share buttons: Twitter/X, WhatsApp, Facebook, LinkedIn — **no auth required** (frictionless virality)
- "Email a friend" button — **auth-gated** (triggers signup modal if not logged in)
- "Copy link" button
- Integrated into `app/results/[id]/page.tsx` as a flex layout with Best Answer card

**Solution — Auth gate modal:**
- `components/auth-gate-modal.tsx` — single modal with signup/login toggle
- Signup fields: First name, Last name, Age range (dropdown: 18-24 through 65+), Email, Password (6 char min)
- Login fields: Email, Password
- Password hashing: PBKDF2 with SHA-512, 100k iterations, random 16-byte salt (`lib/auth.ts`)
- User stored in `localStorage` as `wai_user` JSON — persists across sessions
- Registration auto-subscribes user to QOTD email list via `email_signups` table
- Welcome email + admin notification sent on registration

**API routes:**
- `app/api/auth/register/route.ts` — validates all fields, checks for existing email, hashes password, inserts profile, upserts into email_signups, sends welcome + admin notification
- `app/api/auth/login/route.ts` — validates credentials with timing-safe comparison, returns user object

**Migration:** `supabase/migrations/010_user_profiles.sql` — `user_profiles` table with `id`, `email` (unique), `first_name`, `last_name`, `age_range`, `password_hash`, `salt`, `created_at`. RLS enabled, service-role only access.  
**Status:** ✅ SQL ran, deployed

## Architecture Decisions

1. **Social shares are NOT auth-gated.** This was a deliberate choice — they cost nothing and drive free virality. Gating them would kill the growth loop.
2. **Email-a-friend IS auth-gated.** Someone willing to type a friend's email is high-intent. This is the right moment to capture profile data.
3. **No Supabase Auth.** Using custom PBKDF2 hashing instead. Simpler, no extra dependencies, no auth session management overhead for v1. User is stored client-side in localStorage.
4. **QOTD generates daily, emails every 3 days.** The daily generation feeds the site's QOTD banner. The 3-day email cadence prevents list fatigue and reduces Promotions folder risk.
5. **Email style is intentionally minimal.** No tables, no background colors, no branded header. This is optimized for Gmail Primary inbox placement, not visual branding.

## Known Limitations / Future Work

- **No password reset flow yet.** Would need a reset token email via Resend.
- **No server-side session validation.** Auth is client-side localStorage only. Fine for v1 share gating. Would need JWT or session cookies for any server-side protected features.
- **Gmail Promotions fix needs monitoring.** Gmail's classifier learns over time. If still landing in Promotions after 2-3 sends, next lever is changing `from` name from "Kelly at AskWomensAI" to just "Kelly".
- **how-it-works page** still has its own custom header/footer — could be migrated to `InteriorHeader` for consistency but wasn't touched since it has different nav links.
- **Unsubscribe page** (`/unsubscribe`) is referenced in emails but may not exist yet — needs to be built.
